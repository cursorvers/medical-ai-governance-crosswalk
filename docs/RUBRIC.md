# Requirement Rubric

本資料は法的助言ではない。各セルは原文確認の入口であり、実務判断では必ず公式原文と専門家確認を優先する。

## 判定軸

要求強度は次の 3 軸で判定する。

1. 拘束力: 法令、通知、承認・認証要件、契約上の義務として課されるか。業界慣行、解説、参考情報、政策方針にとどまるか。
2. 対象主体: 開発者、製造販売業者、医療機関、医療者、使用者、規制当局など、誰に課される要求かが明示されているか。
3. 検証可能性: 条文、節、表、箇条書きで確認できるか。周辺記述からの推察、二次資料、一般論に依存していないか。

## Requirement Levels

### `must`

`must` は、法的義務、承認・認証・申請上の要件、又は公的通知で対象主体に明示的に課された要求に使う。

典型語:

- 英語: `shall`, `must`, `required`, `is required to`
- 日本語: `しなければならない`, `必要がある`, `対象となる`, `規制対象`, `必須`

判定条件:

- 拘束力が強い。
- 対象主体が明確である。
- 原文の節・段落で直接確認できる。

### `should`

`should` は、義務ではないが、公式文書が推奨、望ましい対応、留意事項、ベストプラクティスとして明示している場合に使う。

典型語:

- 英語: `should`, `recommended`, `encouraged`, `expected to`
- 日本語: `望ましい`, `推奨`, `重要である`, `留意すること`, `考慮すること`

判定条件:

- 対象主体への行動期待がある。
- ただし法的義務又は承認要件としては明示されていない。
- 原文の節・段落で確認できる。

### `mention`

`mention` は、論点に触れているが、対象主体への要求としては弱い、又は制度紹介・参考情報にとどまる場合に使う。

典型語:

- 英語: `may`, `can`, `available`, `refer to`, `mentions`
- 日本語: `言及あり`, `参照`, `情報提供`, `可能性がある`, `紹介`

判定条件:

- 論点との関係は確認できる。
- 具体的な義務・推奨・検証項目までは示されていない。
- 要求主体が不明又は限定的である。

### `none`

`none` は、原文を確認した範囲で、その論点を扱っていない場合に使う。

条件:

- 公式原文又は対象節を確認済みである。
- 該当論点に関係する記述を確認できなかった。
- `notes_ja` に「確認範囲」を簡潔に残すことが望ましい。

## 空欄 3 状態

空欄は使わない。扱いなし、未評価、出典未確認は必ず区別する。

`none`: 原文確認済みで、該当論点を扱っていない。

`not_assessed`: 原文は存在するが、本資料の当該作業では評価を完了していない。Lane 分担、時間制約、節番号未確定などが理由の場合に使う。

`source_unavailable`: 公式原文にアクセスできない、撤去されている、リンク切れ、又は権限上確認できない。代替 URL を使う場合も、元 URL が確認不可であることを `notes_ja` に明記する。

## 実例

`shall`、`required`、`必須`、`必要がある` と書かれ、対象主体と行為が明示されている場合は `must`。

`should`、`recommended`、`推奨`、`望ましい` と書かれ、行動期待がある場合は `should`。

`may`、`available`、`参照`、`言及あり` のように制度や情報の存在だけを示す場合は `mention`。

原文を確認して該当論点がない場合は `none`。まだ読めていない場合は `not_assessed`。公式原文が取得できない場合は `source_unavailable`。
